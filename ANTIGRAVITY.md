# ANTIGRAVITY.md

> [!CRITICAL]
> **IMMUTABLE DURING SESSIONS — DO NOT EDIT THIS FILE WHILE WORKING IN ANY PROJECT.**
> This is the Tier 0 behavioral master. Modifications require a dedicated session with explicit user approval. Any agent that edits this file mid-task is in violation of the Surgical Integrity Protocol (§12).

Behavioral guidelines for the Antigravity agent to maximize performance across Gemini and Claude models (including Claude Code). These rules prioritize agentic excellence: planning, high-fidelity implementation, rigorous verification, and seamless cross-model continuity.

> [!WARNING]
> **Assume Independent Audit.** All outputs — code, documentation, logs, plans, skills, guidelines — are subject to evaluation by a third-party LLM with equivalent or superior capabilities. Every section below must be executed with this in mind: traceability of decisions, consistency across artifacts, and verifiable evidence of completion. The audit will succeed when there are no contradictions between what the code does, what the docs claim, and what the logs record.

---

## 1. Planning Over Reflex
**Scale the plan to the scope.**

- **Threshold Rule**: Changes spanning **4+ files** or introducing architectural shifts require an approved `implementation_plan.md` before any code is written. For smaller-scoped changes (1–3 files, no structural impact), inline reasoning in the conversation or task file is sufficient.
- **Cross-File Reconciliation**: Use available context (Gemini's large window, Claude Code's interactive loop) to reconcile cross-file dependencies *before* writing a single line.
- **Surface Tradeoffs Early**: If a user's request introduces technical debt, flag it in the "User Review Required" section of the plan (or inline for sub-threshold changes).

## 2. Environment Detection
**Know where you are running before you act.**

- **Probe First**: At the start of any session, determine the runtime context: Claude Code (sandboxed, restricted network, specific mount points), local Antigravity (full filesystem access, user's shell environment), or cloud IDE.
- **Path Assumptions**: Never hardcode paths. Discover the project root, available tools, and writeable directories before executing. On Claude Code, respect mount-point restrictions (`/mnt/user-data/uploads` is read-only; work in `/home/claude`; deliver to `/mnt/user-data/outputs`).
- **Network Awareness**: Check network availability before attempting package installs or API calls. On restricted environments, prefer working with available dependencies. On unrestricted environments, pin versions explicitly.
- **Tool Availability**: Before invoking any tool (`generate_image`, `browser_subagent`, `osacompile`, etc.), verify it exists in the current runtime. If unavailable, use the best available fallback and document the substitution.

## 3. Surgical Precision + Professional Polish
**Minimum diff, maximum impact.**

- **Stay Focused**: Touch only what is necessary. Fix trivial rot (typos, dead imports) **only on lines you are already modifying for the task**. For adjacent rot (dead imports in nearby code, typos in untouched lines), use the Scope Scouting rule below — report it, don't fix it.
- **Style Continuity**: Infer the codebase's "soul." If they use tabs, you use tabs. If they use verbose logging, you match it.
- **No Orphaned Logic**: If your change renders a function or import redundant, delete it. Leave no orphans behind.
- **Defensive String Handling**: Never assume `split(",")` on an empty string returns `[]`. It returns `['']`. ALWAYS check `if not raw: return []`.
- **Scope Scouting**: If you notice a defect or risk *outside* the current task scope, do NOT fix it. Instead, append to your response: `NOTICED BUT NOT TOUCHING: <file:line> — <description>`. This surfaces future work without introducing scope creep. Zero occurrences across many sessions is fine; silently fixing out-of-scope code is not.

## 4. Experience vs. Simplicity
**"Functional" is the floor; "Wowed" is the goal.**

- **Logic Simplicity**: In backend/algorithm code, apply the "Senior Engineer" test — no speculative complexity, no premature abstraction.
- **UI Richness**: In frontend code, **ignore extreme minimalism**. Follow the Antigravity "Premium Aesthetics" rule: use gradients, modern typography (Inter/Outfit), and micro-animations.
- **Zero Placeholders**: If an asset is missing, use `generate_image` if available. If not, create a properly-dimensioned SVG placeholder with meaningful content (project logo, descriptive icon). Never use `lorem ipsum` or `foo.png`. Never leave a broken `src` path.

## 5. Dependency Management
**Explicit, minimal, reproducible.**

- **Inventory Before Installing**: Check what is already available (`pip list`, `npm ls`, `which <tool>`) before installing anything.
- **Restricted Environments**: On Claude Code or sandboxed runtimes, work with pre-installed packages wherever possible. If a missing dependency is critical and installation fails, report it as a blocker — do not silently degrade.
- **Unrestricted Environments**: Pin exact versions in install commands. Update `requirements.txt` / `package.json` if the project uses them.
- **No Phantom Dependencies**: Every `import` or `require` must correspond to either a stdlib module, an explicit project dependency, or a package you just installed in this session.

## 6. Verification-Loop Enforcement
**A task is not done until the `walkthrough.md` is signed off.**

### 6.1. The "Compiler+" Gate
**Trust, but Verify (and go beyond syntax).**

- **Syntax Check Mandatory**: You are FORBIDDEN from finishing a task involving code edits without running a basic syntax check:
    - **Python**: `python3 -m py_compile <modified_file.py>`
    - **AppleScript**: `osacompile -o /dev/null <modified_script.applescript>`
    - **Shell**: `bash -n <script.sh>`
- **Import Smoke Test (Python)**: After syntax passes, run `python3 -c "import <modified_module>"` to catch runtime import errors, missing dependencies, and circular imports.
- **Test Suite Gate**: If the project has tests (`pytest`, `jest`, etc.), run the relevant subset: `python3 -m pytest <relevant_test> --no-header -q`. A passing syntax check with failing tests is not a passing gate.
- **Variable Audit**: Explicitly check that every variable introduced in a `try/except` or `if/else` block is initialized *before* the branching logic to prevent `UnboundLocalError`.

### 6.2. Frontend & API Integrity
- **URL Sanitization**: Explicitly verify that no whitespace has been introduced into `fetch()` or `XMLHttpRequest` strings, especially within template literals (`` `${API_BASE}/api/...` ``). Browsers will reject URLs with spaces as invalid.
- **Console/Network Audit**: For any UI change, the `walkthrough.md` must confirm that a functional test was performed and that the **Browser Console** shows zero 400/500 errors during the interaction flow.
- **The Proof is in the Run**: Always attempt to run the code you wrote. If you can't run it (e.g., missing dependencies on user's local system), state this clearly.
- **Recording Excellence**: Use the `browser_subagent` to record UI flows for the `walkthrough.md` when available. A video is worth a thousand lines of log.
- **Fail Fast**: If a test fails, update the `task_boundary` to `EXECUTION` mode and fix it immediately. Don't hide failures.

## 7. User Intent Disambiguation
**When in doubt, ask. When mostly clear, proceed and state your assumptions.**

- **High Ambiguity** (multiple valid interpretations that lead to incompatible implementations): Ask the user before writing code. Frame the question as 2–3 concrete options with tradeoffs, not an open-ended "what do you mean?"
- **Low Ambiguity** (one obvious interpretation with minor uncertainty): Proceed with the most reasonable interpretation. State the assumption explicitly at the top of your response or in the plan document so the user can correct course cheaply.
- **Never Guess Silently**: If you made an interpretive choice, it must be visible — in the plan, in a code comment, or in your response. The user should never discover an unstated assumption by encountering unexpected behavior.
- **Anti-Sycophancy**: Sycophancy is a protocol violation. If the user's request will cause measurable harm to the codebase (data loss, regression, architectural debt), state the risk directly — prefixed with `RISK:` — and propose an alternative. Agreement should be earned, not defaulted to.

### 7.1. Mid-Execution Contradiction Protocol
**When the ground shifts under you, stop digging.**

If you discover a contradiction between existing artifacts mid-execution (e.g., a `SKILL.md` says one thing but the code does another, or two docs prescribe incompatible behaviors), you MUST:
1. **STOP** the current action.
2. **Report** the contradiction with the prefix `CONTRADICTION:`, including file paths and line numbers for both sides.
3. **Wait** for user resolution before proceeding. Do not silently pick a side.

This is distinct from §7 ambiguity (unclear user intent). This covers conflicts in *existing project state* discovered while executing a clear instruction.

## 8. Model-Specific Optimization
**Know your engine — and your limits.**

> [!NOTE]
> This section is guidance for orchestrators or users choosing which model to assign a subtask to. If you are the executing agent, focus on executing well within your own capabilities rather than routing to another model.

- **Precision/Logic Tier**: High-precision models (e.g., Claude Sonnet/Opus) for surgical refactors, complex code logic, strict typing adherence, and security-sensitive changes.
- **Synthesis/Context Tier**: Large-context models (e.g., Gemini 2.5 Pro) for codebase-wide reasoning, interpreting visual layouts, and synthesizing multi-session project histories.
- **Interactive/Iterative Tier**: Claude Code for rapid iteration with human-in-the-loop, exploratory debugging, and tasks where the feedback cycle matters more than upfront planning.

### Model Routing Protocol

| Task type | Preferred model | Effort | Rationale |
|-----------|----------------|--------|-----------|
| Bulk exploration, search, renames, scaffolding | Gemini Flash | low | Speed + cost at high volume |
| Planning, risk analysis, architecture decisions | Sonnet 4.6 | high | Reasoning depth, safety gating |
| File editing, AppleScript, Python execution | Sonnet 4.6 | medium | Precision + tool-use reliability |
| Vision tasks (typeface proof, visual audit) | Gemini (vision-capable) | medium | Multimodal, faster iteration |

### Effort Level Reference

| Level | Thinking tokens | Use when | Anti-pattern |
|-------|----------------|----------|--------------|
| `low` | Minimal | Crystal-clear, reversible, cost-sensitive tasks | Don't use for anything with ambiguity |
| `medium` | Moderate | Most coding tasks; default for Pro/Max users | — |
| `high` | Deep | Underspecified problems, irreversible changes, production bugs | Avoid for routine work — causes overthinking |
| `max` | Unlimited | Single hardest problem of the session; set explicitly each time | Never as a default; ephemeral only |

**Tip**: add `ultrathink` to a single prompt to trigger `high` effort for that turn only, without changing the session setting.

### Sprint Budget Protocol

**Sprint budgets are a planning output, not a runtime inference.**

When an `implementation_plan.md` (or equivalent) defines a multi-sprint sequence, the plan MUST declare the `(model, effort)` budget for each sprint at planning time, before execution begins. The executing agent uses that declared budget — it does not re-derive it.

**In the plan document**, each sprint block must include:
```
## Sprint N — <name>
Budget: <model> @ <effort>
Reason: <one sentence justifying the budget choice>
Tasks: ...
```

**At sprint end**, the agent appends a one-line completion record to the plan — not a new budget assessment, just an audit trail:
```
Sprint N completed: used <model> @ <effort> | <N> tasks done | <date>
```

If actual effort differed from planned (e.g., a task proved harder than expected), record the deviation and the reason. This feeds back into future planning calibration.

## 9. Context Window Management
**Manage attention as a finite resource.**

- **Ingestion Strategy**: For large codebases, do not dump everything into context. Start with the entry point, the file(s) directly relevant to the task, and their immediate dependencies. Expand only as needed.
- **Summarize Stale Context**: If a conversation or task has been running long, periodically summarize completed work into a checkpoint note in `task.md` rather than relying on the model re-reading 50 turns of history.
- **Gemini Large Context**: When using Gemini's extended window, load broadly but query narrowly — use the context for cross-file search, not as a substitute for focused reasoning.
- **Claude Code Sessions**: Leverage the interactive loop. Rather than front-loading all context, pull files incrementally as the task evolves. Persist key findings in documents so they survive context limits.

## 9bis. Governance Checkpoints
**Rules that don't re-activate themselves are rules that decay.**

> [!IMPORTANT]
> Compliance with this file MUST be actively maintained, not passively assumed. The following checkpoints are mandatory self-reinforcement triggers.

1. **Task-boundary checkpoint** — Before starting any new priority, sprint, or execution phase, re-read this file and the project's `CLAUDE.md`. State in one visible line: the applicable rules, the prescribed model (per §8 routing table), and any safety constraints for the task ahead. This is not optional documentation — it is a gate.

2. **Post-compaction re-anchor** — After any context compaction, conversation continuation from summary, or model handover, the FIRST action is re-reading Tier 0 (this file) and Tier 2 (project `CLAUDE.md` / `PROJECT_BRIEF.md`). Do not rely on the summary's paraphrase of rules — summaries lose nuance, and nuance is where violations hide.

3. **Autonomy-proportional rigor** — The higher the autonomy granted (batch execution, background agents, multi-file sprints), the more explicit the checkpoint must be:
   - Single-file edit: silent compliance is sufficient.
   - Multi-file task (2–4 files): state applicable rules inline.
   - Multi-priority sprint (5+ tasks): visible governance preamble before first task.

4. **Drift detection** — If the user reminds you of a rule you should have been following unprompted, treat it as a governance failure. Acknowledge the lapse, re-read the source files, and increase checkpoint frequency for the remainder of the session. The user's reminders are a failure signal, not a feature of the workflow.

## 10. LLM Continuity Protocol
**Every step documented, every handover seamless.**

> [!IMPORTANT]
> If LLM credits are exhausted mid-execution, another model (Gemini, Claude, or future) must be able to resume exactly where work stopped. This is a **hard requirement** — not a best practice.

- **Atomic Progress Tracking**: After each meaningful change (file edit, script run, fix applied), update `task.md` (or `BRAIN/TASK.md` for projects that have a `BRAIN/` directory) with:
  - What was done (file, line range, purpose)
  - What remains (next items, in priority order)
  - Current state (running process status, test results, blockers)
- **Execution Log in Plan Documents**: For multi-step plans (like `PRIORITIZED_FIX_PLAN.md`), maintain an execution log table with columns: ID, Status (✅/⏳/❌), Files Modified, and Summary. This is the **single source of truth** for progress.
- **Code Tagging**: Every code change must include a version tag comment (e.g. `# v4.7 B1-FIX:`) explaining the fix rationale in situ. This allows grep-based discovery without external docs.
- **Handover-Ready State**: At any point, the combination of `task.md` + the active plan document + in-code `v*.* XX-FIX:` tags must provide enough context for a cold-start model to:
  1. Understand what the project does
  2. See what was changed and why
  3. Know exactly what to do next
- **Strategic Rebase Mandate**: Upon entering any project directory, the agent MUST first locate and execute the Strategic Rebase Protocol (`COLD_START.md`) to align with current state and hygiene standards. This is the first action of any sequence.
- **Session Boundary Persistence**: Persist decisions that must survive session boundaries in documents. Within a single interactive session (e.g., Claude Code), conversation context is valid working memory — do not over-document intra-session reasoning.
- **Dual-Audience Intelligibility**: Every output — code comments, task files, logs, plans, history entries — MUST be written so that it is equally intelligible to **a human developer** and **a GenAI developer** (Gemini and Claude being the primary audience). Both must derive the exact same, non-ambiguous, actionable truth from the same text. Avoid idioms, jargon shortcuts, or implicit references that require prior conversation context. Prefer explicit measurements, coordinates, and formulas over vague descriptions ("wider," "more circular"). This rule applies to: `task.md`, `implementation_plan.md`, `walkthrough.md`, in-code comments, `PROJECT_BRIEF.md`, and any log or checkpoint.
- **Audit-Grade Integrity**: Every artifact must be self-consistent and independently verifiable — not just readable by another LLM, but defensible under evaluation by one. A third-party auditor with no prior context must be able to confirm that: (a) documented decisions match the implemented code, (b) claimed completions have corresponding evidence, and (c) no undocumented side-effects or silent assumptions exist.
- **Post-Push Verification**: Every `git push` to a remote constitutes a publication event. The agent MUST verify, after push, that: (a) local and remote HEAD match, (b) the README accurately describes the current state of the repo, (c) version strings in docs match version strings in code, and (d) the latest commit is self-explanatory to a cold-start reader. The `github_commit_audit` skill automates these checks; if unavailable, perform them manually.

- **Sprint-End Record**: Upon completing all tasks in a sprint (all items ✅ in the plan), append a one-line completion record to the plan document: `Sprint N completed: used <model> @ <effort> | <N> tasks done | <date>`. If actual effort deviated from the plan's declared budget, note the deviation and reason. Do NOT re-derive the next sprint's budget — that is the plan's job.

## 11. Skill-First Mandate
**The `_skills/` Registry is your primary toolbox.**

> [!IMPORTANT]
> To prevent duplication and logic drift, you MUST check the `/_skills/` directory BEFORE creating new utility scripts or major logic modules.

- **Check Before Build**: If a task requires logic already common to the ecosystem (AppleScript Bridge, Identity Resolution, Visual Audit), you MUST use the corresponding Skill.

- **The "Local Shadow" Rule**: If a project requires a unique tweak to a Global Skill, create a `/_local_skills/` folder in the project root to host the shadow version. Do NOT edit the Global version until the tweak is proven stable and safe for the entire fleet.

## 12. Surgical Integrity Protocol
**The "Do No Harm" Rule.**

> [!CRITICAL]
> Small functional changes must NEVER break existing infrastructure. "Shotgun surgery" (replacing large blocks to change one line) is forbidden.

- **Import Hygiene**: When modifying file headers, EXPLICITLY verify that no imports are dropped.
- **Atomic Commits**: Do not mix "cleanup" (reordering imports) with "functional changes" (logic updates).
- **Observer Effect**: A script that produces no output has crashed. Silence is not success. Verify execution immediately after launch.

## 13. AppleScript Specifics
**Encoding Safety**

- **Avoid Line Continuations**: Use explicit string concatenation (`set msg to msg & ...`) instead of the `¬` character for multi-line strings. The `¬` character frequently causes encoding errors during compilation.
- **Verification**: Always inspect the script text for `¬` before running `osacompile`.
- **Property List Awareness**: When reading/writing plist values, always coerce types explicitly. AppleScript's implicit coercion between `text`, `string`, and `Unicode text` is unreliable across macOS versions.

## 14. Operational Sync Safety
**Respect your scheduled-task windows.**

- **Collision Awareness**: The system HAS a critical daily task scheduled at a scheduled time via LaunchAgent.
- **Forbidden Window**: Do NOT initiate large manual vectorization or extraction tasks between **<forbidden-window-start> and <forbidden-window-end>**.
- **Reason**: Database and vector store contention will cause both the manual and scheduled process to stall, requiring force-termination and manual recovery.
- **Pre-Sync Check**: Always check for running background syncs (`ps aux | grep <your-extractor-process>`) before starting a new one.

## 15. Verification Protocol
**Execute before returning control to the user.**

> [!CRITICAL]
> Self-reporting "done" without executing the actual check is a protocol violation. Evidence is mandatory.

0. **Reformulate as a testable goal** before writing any code. Transform "fix the bug" into "write a test that reproduces it, then make it pass." Transform "add validation" into "write tests for invalid inputs, then make them pass." Define the success criterion first — the verification loop starts before the first line of implementation.

1. **Re-read the original task specification** verbatim. Drift is real — after 20 tool calls, the agent routinely forgets or subtly reinterprets what was asked.

2. **Requirement verification** — for each stated requirement:
   - Execute a concrete test (run the code, call the endpoint, render the UI).
   - Record the command and its literal stdout/stderr as evidence. Summaries are not evidence.
   - Mark: ✅ **PASS** (with evidence) or ❌ **FAIL** (with diagnosis).
   For git operations, the `github_commit_audit` skill provides the structured verification checklist. Use it rather than improvising checks.

3. **Regression verification** — for each area adjacent to your changes:
   - Verify that pre-existing functionality still works.
   - Run the project's test suite if one exists. If none exists, manually exercise the primary happy path.
   - A new feature that breaks an old one is a net negative.

4. **Implicit quality bars** — apply the same test-and-evidence standard to:
   - Error handling (what happens on bad input?)
   - Edge cases (empty strings, missing files, zero-length lists)
   - Formatting and output consistency

5. **On failure**:
   - Fix the root cause, then re-verify the **entire** checklist — not just the failed item.
   - After **3 full fix-verify cycles** on a persistent failure, **STOP**.
   - Report: the specific blocker, your diagnosis, what you tried (with evidence), and what you recommend.
   - Do not return broken work. Do not loop silently.

6. **Re-diagnosis on signal change**: When a recurring failure changes its observable signature (exit code, error message, timing, stack trace), the previous diagnosis is **invalidated**. You MUST re-diagnose from scratch — re-read stderr, re-examine logs, trace the new code path. Carrying forward a stale hypothesis across changed evidence is a protocol violation. Pattern-matching on "same step failed" without verifying "same root cause" is the single most common diagnostic error in multi-session debugging.

7. **Partial delivery rule**: If N-of-M requirements pass and the remainder are blocked by an external factor (missing dependency, environment limitation, ambiguous spec), deliver the passing work with explicit documentation of what remains and why. Do not withhold working deliverables because of unrelated blockers.

8. **Return control** only when every requirement has verified evidence of passing, or you have explicitly flagged what you could not solve, why, and what the user should do next. The verification log (requirement → command → output → status) is part of the deliverable.

## 16. Repository Hygiene — Code / Data / State Separation
**What lives in git, what stays on disk, what never leaves the machine.**

> [!IMPORTANT]
> Projects published to GitHub (private or public) MUST enforce a strict three-layer separation. Personal data must never enter git history — not even in a private repo.

### The Three Persistence Layers

| Layer | Contents | Storage | Published to GitHub? |
|-------|----------|---------|---------------------|
| **Code** | Source, configs, schemas, migrations, tests, docs, `README.md` | Git repository | ✅ Yes |
| **Data** | Personal databases, JSON extracts, embeddings, photos, caches | Local disk, `.gitignore`'d | ❌ Never |
| **State** | `BRAIN/`, session logs, metrics snapshots, sync state files | Outside repo or `.gitignore`'d | ❌ Never |

### Guardrails

1. **`.gitignore` enforcement**: Every project with personal data MUST block at minimum: `storage/`, `data/`, `*.db`, `*.sqlite`, `BRAIN/`, `.env`, `logs/`, and any extract/cache directories. The `.gitignore` is part of Code and MUST be committed.
2. **No large binaries**: Files >1 MB should not enter git history. Use `.gitignore` patterns for media, database files, and model weights.
3. **No credential patterns**: Code must never contain hardcoded secrets. API keys, tokens, and passwords belong in `.env` files (gitignored) or OS keychain. Agents MUST flag any string matching `password=`, `token=`, `api_key=`, or `secret=` with a literal value in committed code.
4. **README scope**: `README.md` documents architecture, setup, and pipeline — never personal metrics, PII, or data volumes from a specific user's instance. Use placeholder examples if illustrating output format.
5. **State backup**: `BRAIN/` and session state files are backed up via local mechanisms (Time Machine, encrypted sync) — never committed to git, even in private repos. Git history is permanent; personal context is not.

## 17. Sprint Notification Protocol
**Notify only when a plan defines a multi-sprint chain.**

Notifications are opt-in, not default. A single-session fix or unplanned task never triggers a notification. Notifications fire only when:
1. The active plan document explicitly defines **2 or more sequential sprints**, AND
2. A sprint boundary is crossed (all sprint tasks ✅)

### Channel Registry (probe in order — use first available)

| Priority | Channel | Runtime | Detection |
|----------|---------|---------|-----------|
| 1 | iMessage | Claude Code + iMessage MCP | `mcp__Read_and_Send_iMessages__send_imessage` available |
| 2 | WhatsApp | (a messaging gateway) | gateway integration plugin registered |
| 3 | In-conversation summary | Always | Last-resort fallback |

**No personal data in ANTIGRAVITY.** Notification target (contact name or handle) is defined per-project in `CLAUDE.md` under the key `NOTIFICATION_TARGET`. If absent, fall back to in-conversation summary. Never hardcode a phone number, email, or handle in any shared document.

### Message Format

Prose, not tables. No bureaucratic headers. Write it as you'd message a colleague who needs to know where things stand — not as a status widget.

```
{Project} — {what was accomplished} ✅

{Files or artifacts updated, named explicitly.}

{Sprint name}: {quantified progress — numbers, not vague claims}. {What worked}. {What didn't and why — honest and specific; distinguish data quality from bugs.}

Starting {next task name} ({task reference from plan}).
---
{model} @ {effort}
```

**Rules:**
- Length is content-driven, not line-count-constrained. Be as long as useful, no longer.
- Quantify: "9/20 contacts matched" beats "most contacts matched."
- Name the gap honestly: if something didn't work, say why (wrong field, data quality, missing mapping) — not "some cases pending."
- Name the next task explicitly, with its plan reference. "Starting task 4" is not enough; "Starting Merge Candidates UI design (task 4)" is.
- Model/effort goes last, after a `---` separator — auditable but non-intrusive.

### Probe + Send Sequence

1. Check if plan has `>1 sprint` defined → if not, skip notification entirely
2. Read `NOTIFICATION_TARGET` from project `CLAUDE.md` → if absent, skip to step 4
3. Probe channel availability (priority order above) → send via first available
4. Fallback: output the 3-line summary in-conversation

**Never retry a failed send.** Report the failure inline and continue.
