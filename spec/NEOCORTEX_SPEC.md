# NEOCORTEX_SPEC — Project State Layer (L3)

**Status: v1.0 — RATIFIED 2026-06-10** — companion to
`PAICodeConstitution-2026.md` Art. 6; binding per its transitional clause.
This spec itself is **L4 reference data**: amended freely via git
(Constitution layer table), no Constitutional Review needed. Supersedes the unspecified `BRAIN/` convention (2025)
**project by project**; non-migrated projects keep their `BRAIN/` until their
migration session.

**Purpose.** The NEOCORTEX is a project's continuity store: everything a
cold-start agent (or human) needs to resume work, bounded so it can actually
be read. The 2025 BRAIN failed by unbounded growth (520 KB journals, 242 KB
status files — larger than a context window) and by having no schema (17+
ad-hoc document genres across 18 projects). This spec fixes both.

**The project unit.** One NEOCORTEX per project; a *project* is the folder
that owns its own `CLAUDE.md` (or harness equivalent). Multi-module
workspaces (e.g., a parent with modules 01..09) follow ownership: a module
with its own `CLAUDE.md` gets its own NEOCORTEX; modules without one share
the parent's. The `governance/` program itself is exempt: its `PLAN_*` doc
serves as its STATUS until rollout completes, after which it gets a NEOCORTEX.

---

## 1. Directory layout

```
<project>/NEOCORTEX/
  MANIFEST.json          # mandatory — machine-readable index, archive declaration
  STATUS.md              # mandatory — bounded rebase block, OVERWRITTEN in place
  JOURNAL.md             # mandatory — current quarter only, prepend-newest
  <GENRE>_<slug>_<YYYY-MM-DD>.md   # typed documents, see §3
  _archive/              # frozen history — never deleted, always declared
```

**State vs. git (Constitution Art. 5).** Published history = any repository
with a remote. In a project whose repo has a remote, `NEOCORTEX/` is
`.gitignore`'d. A project may track its NEOCORTEX in a **local-only** repo by
explicit decision recorded in its `CLAUDE.md`. Either way, state is backed up
by local mechanisms (Time Machine, encrypted sync), never pushed.

## 2. The three mandatory files

### MANIFEST.json
Regenerated by the validator (`neocortex_manifest.py`, an L1 deliverable —
rollout plan Phase 4) whenever files are added or retired; hand-editing is
permitted but **the directory is the truth and the manifest must match it** —
on disagreement, the directory wins and the manifest is regenerated. The
validator runs in the session-close hook (Phase 3), not just at audits.
**The first entry always declares the archive** — "we know it exists."

Required fields: `spec`, `project`, `updated`, `archive{exists, span,
contents, note}`, `files[]` with `{file, genre, status, hook}` — every `.md`
in the directory except JOURNAL.md and STATUS.md must appear in `files[]`;
`genre` must be one of the six in §3. MANIFEST.json is the single sanctioned
non-markdown file in the NEOCORTEX (exhaustive exemption to the §3 data rule).

```json
{
  "spec": "NEOCORTEX_SPEC v1.0",
  "project": "<name>",
  "updated": "YYYY-MM-DD",
  "archive": {
    "exists": true,
    "span": "2026-03 .. 2026-06",
    "contents": "pre-migration BRAIN (journal 520KB, 31 plans, 2 incidents)",
    "note": "frozen verbatim at migration; mine on demand, never delete"
  },
  "files": [
    {"file": "PLAN_x_2026-06-10.md", "genre": "PLAN", "status": "ACTIVE", "hook": "one-line summary"}
  ]
}
```

### STATUS.md — the bounded rebase block
- **Hard bound: 150 lines.** Overwritten in place every session — never appended.
  If it doesn't fit, it isn't status; it's history → JOURNAL.
- Required sections, in order:
  1. `## What this project is` (≤5 lines)
  2. `## Current phase` (one line + date stamp)
  3. `## Invariants` (**≤15**, numbered; cite the INCIDENT/DECISION doc that
     created each — citations may point into `_archive/`; overflow beyond 15
     is consolidated into a `PROTOCOL` doc which the list then cites)
  4. `## Next actions` (≤5, priority-ordered, each independently actionable cold)
  5. `## Pointers` (active plans, key data files, test command; the
     notification policy lives in `CLAUDE.md` only — STATUS may point to it,
     never restate it)

### JOURNAL.md — rotated, never unbounded
- Prepend-newest session entries: what was done (files, ranges, purpose),
  what remains, current state (processes, test results, blockers). Change
  rationale lives **here and in commit messages** — the 2025 in-code
  version-tag convention (`# vX.Y FIX:` comments) is retired; existing tags
  are grandfathered, never extended.
- **Rotation rule: at each quarter boundary, or upon exceeding 64 KB,**
  whichever comes first → move to `_archive/JOURNAL_<YYYY>Q<n>.md` and leave a
  ≤10-line digest at the top of the fresh JOURNAL. The size trigger is
  checked by the session-close hook (Phase 3); any agent may perform the
  rotation.

## 3. Document genres — exactly six

| Genre | Holds | 2025 genres it absorbs |
|---|---|---|
| `PLAN` | Multi-step work with task-class budgets (Art. 9) + execution log. Required for work spanning ~4+ files or shifting architecture (Art. 13). **ACTIVE plans are bounded: ≤300 lines**; overflow goes to an `_archive/` continuation noted in the plan | PLAN, SPEC, DESIGN, IMPL |
| `AUDIT` | Point-in-time evidence-based assessment, incl. verification walkthroughs | AUDIT, ANALYSIS, REVIEW, COMPARE, **WALKTHROUGH** |
| `INCIDENT` | What broke, why, and the L1/L2 rule it produced | INCIDENT |
| `DECISION` | A choice, its alternatives, and its rationale | DECISION, REFLECTION, ADVISORY |
| `NOTE` | Anything ephemeral that earns persistence | NOTE, SESSION, HANDOVER, EXPLAINER |
| `PROTOCOL` | Project-specific operating procedure or consolidated invariants | PROTOCOL, STANDARDS, CONVENTIONS |

- Naming: `GENRE_slug_YYYY-MM-DD.md`. Header line 1 carries
  `Status: ACTIVE | DONE | SUPERSEDED-BY <file> | ARCHIVED`.
- **PLAN validation gate**: when a PLAN is drafted, the drafting agent
  **proposes an adversarial review before validation, stating reasons** —
  the expected failure modes for *this* plan (omissions vs. predecessor
  artifacts, internal contradictions, unenforceable mandates, unnamed
  deliverables, lost requirements). When the plan migrates or supersedes
  prior rules, the review checks all **four loss modes**: content lost,
  scope narrowed, **timing shifted** (a "before" that became an "after"),
  and force weakened (a MUST that became a log entry) — a relocation claim
  is valid only if all four are preserved. (Both 2026-06-10 semantic losses
  were modes 2–4; the content-only review traced them as COVERED.) The user may waive the review; the
  proposal itself is mandatory. The reviewer, when run, must not be the
  drafting model (routing_policy `verifier-not-drafter`). Lesson source: the
  2026-06-10 constitution review round — 10 HIGH findings caught between
  draft and ratification, several invisible to the drafter.
- `DONE`/`SUPERSEDED` files move to `_archive/` at the next session close.
  INCIDENT docs may be archived once their lesson is encoded in L1/L2;
  STATUS invariants may cite them in `_archive/`.
- **No data in the NEOCORTEX** (MANIFEST.json excepted): any other
  non-markdown file, or anything > 64 KB, belongs in the project's `data/`
  directory (Constitution Art. 5). The validator flags violations.

## 4. Cold-start read order

`MANIFEST.json` (≈2 KB) → `STATUS.md` (≤150 lines) → ACTIVE plans (≤300
lines each). With these bounds a typical cold start costs **under 10k
tokens**; a project with several active plans may exceed it — the validator
reports the current cold-start cost so growth is visible, not silent.
JOURNAL and archives are consulted on demand, not by default.

## 5. Migration procedure (one project per session, never fleet-wide)

1. Rename `BRAIN` → `NEOCORTEX` (`git mv` only in repos where state is
   tracked per the §1 Art. 5 decision; plain `mv` otherwise) and apply the
   §1 state-vs-git rule: add `NEOCORTEX/` to `.gitignore` in any repo with a
   remote.
2. Sweep all existing files into `NEOCORTEX/_archive/` **except**: ACTIVE
   plans, INCIDENT docs still uncoded in L1/L2, and PROTOCOL docs still in
   force (rename to spec).
3. Write a fresh bounded `STATUS.md` from current reality (not from the old
   one — that's the point).
4. Start a fresh `JOURNAL.md`; the old journal goes to `_archive/` whole.
5. Generate `MANIFEST.json` (validator) with the archive declaration.
6. Update the project's `CLAUDE.md` (and any `AGENTS.md`/Antigravity rules
   file): read order = manifest → status → active plans; all `BRAIN/`
   references repointed; any `COLD_START.md` retired — its still-true content
   merges into STATUS/PROTOCOL, the file itself goes to `_archive/`.
7. Install the L1 enforcement adapter for this project: `python3 adapters/claude-code/install_adapters.py --apply` (generated local state, gitignored — idempotent, re-run anytime).
8. Evict data files (>64 KB, non-md) to `data/`.
9. Record the migration as a `NOTE_neocortex_migration_<date>.md` with a
   before/after inventory — the audit evidence (Art. 1).

## 6. What stays in the project's CLAUDE.md (not here)

Notification target, cadence, and trigger policy (single home — Art. 10);
domain safety rules; skill versions in use; environment quirks. The
NEOCORTEX holds state; CLAUDE.md holds configuration.
